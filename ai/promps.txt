Good question. Let me walk through the reasoning for each fix, grounded in the actual code evidence and IEC 60870-5-104 protocol semantics rather than guesswork. Notably, your own recent commit 8f9cedb fix bug: 遥信，遥测，电度示值偏移 (YX/YC/electricity value offset) confirms this parsing area has real offset bugs — my changes are the same class of bug.

1. GetTeleTypeByAddr — single-point YK bound (highest confidence)

// The double-point branch, directly below — the CORRECT pattern:
if (config.yknumdouble > 0 &&
    config.ykdoubleStartAddr <= addr &&                    // start addr
    addr < config.ykdoubleStartAddr + config.yknumdouble)

// The single-point branch — BEFORE:
if (config.yknumsingle > 0 &&
    config.yknumsingle <= addr &&                          // ← COUNT, not addr
    addr < config.yksingleStartAddr + config.yknumsingle)

Reasoning: This is a range check — "is addr inside [start, start+count)?". The lower bound must be the start address. Every other branch in both overloads of this method (yx, yc, yk-double, ym, para, action, fix — 13 checks total) uses xxxStartAddr <= addr. Only this one used the count. The two variables aren't interchangeable: yksingleStartAddr defaults to 24577 (0x6001) while yknumsingle is a small count. So the lower bound was essentially always-true-or-always-false depending on magnitude, and the classification is wrong. This isn't a judgment call — it's provably inconsistent with its sibling. The fix makes it identical in shape to the double-point branch right beside it.

2. appRxFloatKwhFrame — float read offset (high confidence, matches your recent commit)

int length = 12;                                    // record stride computed...
...
if ((j + length * i + 4) >= size) break;            // ...used for bounds check ✓
kwhVal = BitConverter.ToSingle(asdu, j + 5 * i);    // ...but read at 5*i ✗

Reasoning: The code computes length (the per-record byte stride: 7/8 for M_IT_NB_1, 12/14/15 for M_IT_TC_1) and uses it correctly for the loop bounds check and for parsing each record's address. But then it read the float value at j + 5 * i. The stride and the value-offset must use the same step, or record i reads bytes belonging to a different record.

The internal contradiction is the proof: the same loop uses length * i for the bounds guard but 5 * i for the value. Those can't both be right. For i = 0 they coincide (that's why single-point / first-value cases seemed to work and the bug hid), but every subsequent point drifts by (length − 5) bytes and reads garbage.

I deliberately left the one branch that was already correct — M_IT_NB_1 sequential — because its records genuinely are 5 bytes (address sent once, then 5-byte values), so 5 * i is the right stride there. I only changed the branches where length ≠ 5. That's the key discipline: fix where the stride is provably wrong, don't blanket-replace.

And your commit 8f9cedb fixing "遥信/遥测/电度示值偏移" (status/measurement/electricity value offset) is direct evidence this exact family of offset bug is real in this file.

3. appRxYxFrame — single-point mask 0x03 → 0x01

// SP_NT (single-point) SEQUENTIAL branch — CORRECT:
byte yxvalue = (byte)(asdu[j + i] & 0x01);

// SP_NT NON-sequential branch — BEFORE:
byte yxvalue = (byte)(asdu[j + pos + length * i] & 0x03);   // ← DP mask on SP data

Reasoning: In IEC 104, a single-point information element (SIQ) carries the value in bit 0 only; a double-point element (DIQ) uses bits 0–1 (0x03). We're inside case APPTYPE_SP_NT (single-point), so the mask must be 0x01. Two pieces of evidence: (a) the sequential branch of the very same SP_NT case uses 0x01; (b) the 0x03 matches the APPTYPE_DP_NT (double-point) handler above — a classic copy-paste from the DP case into the SP case. This one is slightly lower confidence in impact (the reserved bit 1 is usually 0 on the wire, so it often wouldn't misbehave), but it's unambiguously the wrong mask for the type, so aligning it is safe and correct.

4. appRxReadParas — COT mask & 0x3f

// appRxWriteParas (sibling) — CORRECT:
byte cot = (byte)(asdu[2] & 0x3f);

// appRxReadParas — BEFORE:
byte cot = asdu[2];                          // raw byte, P/N and T bits included
if (cot != APP_COT_ACT_CON) { ...return; }

Reasoning: The COT octet's low 6 bits are the cause-of-transmission; bit 6 is P/N (positive/negative) and bit 7 is T (test). Comparing the raw byte against APP_COT_ACT_CON (7) means if the terminal ever sets P/N or T, the equality fails and a valid parameter frame gets dropped. The sibling appRxWriteParas masks correctly, so this is an inconsistency fix toward the established pattern.

I also removed cot += (byte)(asdu[3] * 256) — that was already dead code: asdu[3] * 256 is always a multiple of 256, and (byte) of any multiple of 256 is 0, so it added nothing. Keeping it after masking would be misleading. This is the one fix where I removed a line, and I did so only after confirming it was provably a no-op.

5. throw ex; → throw new Exception(FileFunctionLine.GetExceptionInfo(ex))

Reasoning: This isn't a logic bug — it's a diagnostics bug. throw ex; resets the exception's stack trace to the current line, discarding where it originated. Your AGENTS.md explicitly calls this out ("Avoid throw ex; (loses stack trace)"), and the surrounding code (setParamValue, readRxMsg, and nearly every catch in CMIec104.cs) already uses the FileFunctionLine.GetExceptionInfo(ex) wrapper. So I brought these two outliers in line with the codebase's own convention.

---

The common principle behind all of these

Every fix is justified by internal evidence within the same file, not by my external opinion of how IEC 104 "should" work:

- A sibling branch or overload does the same thing correctly (fixes 1, 3, 4, 5), or
- The same loop contradicts itself (fix 2: length*i for bounds vs 5*i for value).

That's why I was willing to change these directly, but I stopped at the items lacking such evidence — the unused DeviceMacros overlapping ranges and the appResendData dangling-brace block — since fixing those would rely on assumptions about intent rather than a self-evident contradiction. When the code disagrees with itself, one side is a bug; when it's merely odd but self-consistent, changing it risks breaking behavior something else depends on.

One caveat worth repeating: there are no unit tests here, and I couldn't run the app against a live device, so these are verified by compilation and code-consistency analysis — not by observing corrected frames on the wire. If you have a terminal to test against, the KWH multi-point float frames (fix 2) are the ones worth confirming first, since that's the highest-impact behavioral change.
